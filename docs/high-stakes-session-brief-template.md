# High-Stakes Session Brief 模板

date: 2026-07-03
source: Fable 5 Session 3 蒸餾（對照 Session 1/2 的 brief 與實際產出）。
只保留有行為證據的結構元素；被刪掉的儀式列在文末。
搭配 `weak-model-extraction-guide.md` 使用——該文解釋每個模式為什麼有效，
本文只負責「照抄可用」。

使用方式：複製本檔、填掉所有 `[...]`，session 開始時**只貼「開場
Prompt」fence 內的內容**。fence 外的所有段落是給你自己的，模型不會
看到——想約束模型的話，把那句話搬進 fence。

---

## 0. 值得開高風險 session 嗎（先過這關）

一句話回答：**這個任務弱模型會系統性失敗嗎？**
「系統性」指的是失敗源於能力或傾向（順從、看不出設計矛盾、不敢砍東西），
而不是源於缺資訊。缺資訊的任務把資訊補進前置資訊區就好，不用燒高價
session。寫下你的理由：

> [為什麼弱模型做不到；這次 session 的產出會被攤提到哪些未來工作]

## 1. Session 前準備（不花高價 token，自己或便宜模型完成）

- [ ] 相關 repo git 狀態乾淨；驗證命令（如 verify.sh）目前是綠的，
      收尾驗證才有對照基準。
- [ ] permissions 先設好，避免中途打斷連續推理。
- [ ] 填好下方前置資訊區：事實與直覺**分開**，直覺不要偽裝成事實。
- [ ] 想清楚產出位置表（Layered Ownership）：什麼寫到哪、哪裡是禁區。

## 2. 開場 Prompt（直接複製貼上；模型只會看到這段）

```
[一句話任務定位：這個 session 的產出是持久資產（ADR、doctrine、交接
文件、模板），不是一次性修改。]

背景：[兩三行。這個 repo/系統是什麼、產出會影響什麼範圍。]

任務分階段：

1. 【[診斷/Audit/分析]】[讀什麼、回答什麼問題。]
   先報告發現，不要動手改。發現超過 5 個時排出優先序。
   每個發現要指出具體檔案與具體風險。

2. 【[設計/裁決]】針對最高影響的 3-5 個問題提出方案，說明取捨，
   等我確認。[若裁決型任務，加上可檢核的裁決格式：]
   每個對象給「保留／凍結／合併／砍掉」四選一裁決附理由；
   全部「保留」視為未完成任務。

3. 【[落地/交付]】實作確認後的項目，並且：
   - 判斷難、執行機械的部分不要實作——切成獨立 ticket，每張
     自帶驗證步驟，交給便宜模型執行；
   - 跑 [驗證命令] 驗證；
   - 最後寫交接文件到 [位置]：判斷依據、不放心的地方
     （已驗證 vs 推定要分開標注）、給未來較弱模型的注意事項。

產出位置（Layered Ownership，逐項明列）：
- [產出 A] → [路徑]
- [產出 B] → [路徑]
- 禁區：[凍結 repo、別層的文件]。若認為禁區該動，寫成給我的建議，
  不要直接動手。

約束：
- 只動任務需要的檔案；不刪除任何檔案；不建新 repo、新流程、新工具
  （process-vs-work doctrine 適用）。
- 你不確定的地方明說「不確定」，不要編造。
- 裁決前先收集可計數的證據（commit 數、使用次數、被讀取引用次數），
  裁決必須引用數字；沒有數字支撐的判斷標記為直覺。

驗收條件（達不到就是任務未完成）：
- [可檢核的條件，例：每層都有四選一裁決；至少有非「保留」結論；
  交接文件含「不放心的地方」清單]

前置資訊：
- 事實（可採信）：
  - [實際使用情況、已確認的執行狀態、有日期的事件]
- 直覺（請獨立驗證，不要直接採信；對每條給出成立／不成立／部分成立）：
  - [你的懷疑、印象、猜測]
```

## 3. Session 中操作手冊（給你，不是給模型）

- 階段門檻逐項確認才放行；模型跳過門檻直接動手時，打斷並退回。
- 裁決結果全是「保留」或全是好話時，追問：
  **「如果你必須砍掉兩個，你砍哪兩個？為什麼？」**
  排序題繞過順從，不需要模型對抗你。
- 範圍看門：設計階段只要「最便宜可行」，模型開始設計需要維護的
  基礎設施時，引用 doctrine 擋下。
- （選配，視工具而定）plan mode 可作為階段 1-2 的載體；大範圍掃檔
  可派 subagent。這兩項無實證承重，prompt 內的門檻語句才是必要的。

## 4. 收尾

- [ ] 留夠 context 讓模型寫交接文件——對話會結束，檔案不會。
- [ ] 交接文件含「不放心的地方」清單（此清單在 Session 1 後真的被
      後續決策引用過，是模板中證據最強的元素之一）。
- [ ] 用 capture-assistant-session 存進 work-log vault。
- [ ] 砍/併/凍結類裁決由你手動執行；AI 不做批量刪除。

## 明確刪掉的儀式（別加回來）

- 「你是我唯一一次使用最高階模型的機會」稀缺性開場——兩次 capture
  無行為證據；有效成分是「產出持久資產」定位句，已內建在 prompt 首行。
- 「不要客氣，順從對我沒有價值」態度句——結果達標但因果無法歸因；
  已改寫成驗收條件裡的可檢核格式（四選一＋至少非保留）。
- fence 外的「預期交付物與驗收」表——模型看不到 fence 外的東西；
  驗收條件已搬進 prompt 本體，你自己的 checklist 用第 1、4 節。
