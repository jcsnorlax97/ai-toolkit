# 如何從弱模型逼出接近強模型的判斷

date: 2026-07-03
source: Fable 5 Session 3（繼承蒸餾）。每個模式標注驗證狀態：
**已驗證** = 在 Session 1/2 有具體行為證據；**機制可信** = 原理成立但
因果證據薄，用時自己觀察。只收有信心推薦的模式，湊數的已剔除。

## 1. 事實／直覺分離 —【已驗證】

**何時用**：任何你要提供背景資訊的 brief。
**怎麼寫**：前置資訊分兩區——「事實（可採信）」與「直覺（請獨立驗證，
不要直接採信）」，並要求對每條直覺明確給出成立／不成立／部分成立。
**證據**：Session 2 的模型推翻了直覺 (4)（skillops 問題是只寫不讀，
不是沒人填）並發現直覺 (5) 部分過時（dropzone 已存在，沒有順著重建）。
標示「請驗證」給了模型反駁的許可——這對天生順從的弱模型尤其關鍵。

## 2. 先報告、等確認才動手 —【已驗證】

**何時用**：產出不可逆、或判斷比執行更重要的 session。
**怎麼寫**：在 prompt 本體寫「先報告，等我確認後才寫檔案／動手」，
逐項確認後才落地。這是工具無關的寫法；plan mode 只是它的一種載體，
兩者不必疊加（哪個承重無法區分，擇一即可）。
**證據**：兩次 session 都被遵守，裁決逐項經確認才落地。

## 3. 只設計不實作，實作切成獨立 ticket —【已驗證】

**何時用**：判斷難、執行機械的工作。這是強弱分工的切點，在沒有強模型
的時代同樣適用：一個「仔細的 session」做裁決與設計，其他 session 做執行。
**怎麼寫**：「只設計，不實作」＋要求每張 ticket 獨立可做、自帶驗證步驟
（「完成一張驗證一張」）。
**證據**：Session 2 產出 6 張 ticket，事後全部由便宜模型執行成功。

## 4. 強制明說不確定＋交接文件含「不放心的地方」—【已驗證】

**何時用**：每個高風險 session 的收尾。
**怎麼寫**：「你不確定的地方明說『不確定』，不要編造」＋「最後寫交接
文件：判斷依據、不放心的地方、給未來模型的注意事項」。要求模型把
「已驗證」與「推定但未重現」分開標注（如 Session 1 對 tmp-link 根因
的標注方式）。
**證據**：Session 1 交接文件的「Not fully verified」清單事後真的被
引用（memory、Session 3 brief 都指向它）——寫了有被讀。

## 5. 預先指定產出位置與所有權邊界 —【已驗證】

**何時用**：跨 repo／多層 workspace 的任何寫入型任務。
**怎麼寫**：在 prompt 裡逐項列「什麼寫到哪」，並點名禁區（凍結 repo、
別層的文件）。不要假設模型會自己推斷 Layered Ownership。
**證據**：兩次 session 都零跨層寫入；Session 2 連「若認為凍結層該解凍」
都按指示寫成建議而非直接動手。

## 6. 強制非順從產出：可檢核的裁決門檻＋排序追問 —【機制可信】

**何時用**：audit、裁決、review——任何「全部保留」等於失敗的任務。
**怎麼寫**（兩層）：
- 在 prompt 裡放**可檢核**的驗收條件：「至少要有非『保留』的結論」
  「每層給保留／凍結／合併／砍掉四選一」。弱模型對可檢核條件的服從
  遠比對「不要客氣」這類態度指令可靠。
- 若產出仍是全保留，追問：「如果你必須砍掉兩個，你砍哪兩個？為什麼？」
  排序題繞過順從傾向——它不要求模型對抗你，只要求比較。
**誠實標注**：Session 2 結果達標（凍結 skillops、砍兩項、13:1 判定
不健康），但「不要客氣」一句話的因果貢獻無對照組；追問句當時沒觸發
（audit 本來就給了非保留結論）。推薦依據是機制，不是實證。

## 7. 裁決前先收集可計數證據 —【機制可信】

**何時用**：弱模型做價值判斷之前。這是最直接補償判斷力差距的手段：
把「你覺得這層值得留嗎」換成「先數，再按規則裁」。
**怎麼寫**：「裁決前先收集客觀數據（commit 數、真實使用次數、被讀取
引用次數），每個裁決必須引用數字；沒有數字的裁決標記為直覺」。
**誠實標注**：Session 2 的強模型是自發這樣做的（13:1、45 份 capture、
evidence 只被讀過一次），未在弱模型上實測過指令化的效果；但收集本身
是機械工作，弱模型做得到。

## 明確不收錄（有人問起時的答案）

- 「你是我唯一一次使用最高階模型的機會」稀缺性開場——兩次 capture
  均無行為證據；有效成分是「優先產出持久資產」那句，直接寫那句即可。
- plan mode 指引與 Explore subagent 指引——工具綁定、無使用證據；
  模式 2 的「先報告等確認」已覆蓋前者的作用。
